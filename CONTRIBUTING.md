## ü§ù How Can I Contribute?

There are many ways to contribute to this project:

### Reporting Bugs

- Use the GitHub Issues tab to report bugs
- Describe the bug in detail with steps to reproduce
- Include screenshots if applicable
- Mention your environment (OS, browser, Node version, etc.)

### Suggesting Enhancements

- Use GitHub Issues to suggest new features or improvements
- Clearly describe the enhancement and its benefits
- Provide examples or mockups if possible

### Contributing Code

- Fix bugs or implement new features
- Improve documentation
- Add examples or tutorials
- Enhance existing projects

### Improving Documentation

- Fix typos or clarify existing documentation
- Add missing documentation
- Create tutorials or guides
- Translate documentation

## üöÄ Getting Started

1. **Fork the Repository**
   - Click the "Fork" button at the top right of the repository page

2. **Clone Your Fork**
   ```bash
   git clone https://github.com/YOUR-USERNAME/FDP-BATCH7.git
   cd FDP-BATCH7
   ```

3. **Create a Branch**
   ```bash
   git checkout -b feature/your-feature-name
   ```

4. **Make Your Changes**
   - Write clean, readable code
   - Follow the existing code style
   - Test your changes thoroughly

5. **Commit Your Changes**
   ```bash
   git add .
   git commit -m "Add: brief description of changes"
   ```

6. **Push to Your Fork**
   ```bash
   git push origin feature/your-feature-name
   ```

7. **Create a Pull Request**
   - Go to the original repository
   - Click "New Pull Request"
   - Select your fork and branch
   - Fill out the PR template with details

## üìù Contribution Guidelines

### Code Contributions

- **Quality**: Ensure code is clean, well-commented, and follows best practices
- **Testing**: Test your changes before submitting
- **Documentation**: Update relevant documentation
- **Dependencies**: Minimize new dependencies; justify if necessary
- **Compatibility**: Ensure changes don't break existing functionality

### Documentation Contributions

- Use clear, concise language
- Follow existing documentation structure
- Include code examples where appropriate
- Check for spelling and grammar errors

## üé® Style Guidelines

### Code Style

- **JavaScript**: Use ES6+ syntax, follow Airbnb style guide
- **HTML**: Use semantic HTML5 elements
- **CSS**: Follow BEM naming convention or existing patterns
- **Indentation**: Use 2 spaces (no tabs)
- **Naming**: Use descriptive variable and function names

### File Naming

- Use lowercase with hyphens for file names (e.g., `my-component.js`)
- Use descriptive names that reflect the file's purpose

## üí¨ Commit Message Guidelines

Write clear, meaningful commit messages:

### Format
```
Type: Brief description (50 chars or less)

Detailed explanation if necessary (wrap at 72 chars)
```

### Types
- **Add**: New feature or file
- **Fix**: Bug fix
- **Update**: Update existing feature
- **Refactor**: Code refactoring
- **Docs**: Documentation changes
- **Style**: Code style changes (formatting, etc.)
- **Test**: Adding or updating tests
- **Chore**: Maintenance tasks

### Examples
```
Add: User authentication feature

Implemented JWT-based authentication with login/logout functionality.
Includes middleware for protected routes.
```

```
Fix: Navigation menu not responsive on mobile devices

Adjusted CSS media queries and added hamburger menu toggle.
```

## üîÑ Pull Request Process

1. **Ensure Prerequisites**
   - Code follows style guidelines
   - All tests pass
   - Documentation is updated
   - Commit messages are clear

2. **Fill Out PR Template**
   - Describe what changes were made
   - Explain why the changes are necessary
   - Reference related issues

3. **Review Process**
   - Maintainers will review your PR
   - Address any requested changes
   - Be patient and respectful

4. **Merging**
   - PRs require approval from maintainers
   - Once approved, your PR will be merged
   - You may be asked to squash commits
